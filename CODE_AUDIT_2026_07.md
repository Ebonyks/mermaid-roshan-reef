# Mermaid Roshan Reef of Light — Code Construction Audit (July 2026)

*Scope: implementation quality of the shipped GDScript (Godot 4.4, Forward Mobile).
This is a code audit — where things are built well, where they are built poorly,
confirmed bugs with proposed fixes. It deliberately does not propose new features.
Design/content critique lives in GAME_AUDIT_v3_49.md and AUDIT_3_0.md.*

> **Status update (2026-07-11):** the P0 fixes (B1–B4) and P1 fixes (B5–B8) below
> have been applied on this branch. Line numbers in the bug descriptions refer to
> the pre-fix `master` revision. §4 structural items remain open.

Codebase surveyed: `scripts/main.gd` (7,010 lines), `scripts/player.gd` (332),
`scripts/touch_ui.gd` (143), 10 probe/test scripts, vendored `tessarakkt.oceanfft`
addon (disabled at runtime), `project.godot`, 87 MB of assets.

---

## 1. Overall construction

The entire game is procedural: one autogenerated `main.tscn` containing a single
`Node3D` with `main.gd` attached. Every mesh, material, UI panel, minigame, save
system, audio player and cutscene is built in code at runtime. There are no
scene files, no `Area3D`/physics bodies, no `InputMap` actions, no signals between
game systems — everything is distance checks and shared mutable dictionaries
inside one 7,010-line class.

This is an unusual architecture with real advantages at this scale (diffable,
no scene-file merge pain, everything greppable) and it demonstrably works — the
game ships and the headless probe suite can win all five friend games. But it is
also the codebase's biggest liability, detailed in §4.

## 2. Strengths (things done genuinely well)

- **Frame-rate-independent math everywhere.** Damping and smoothing consistently
  use `pow(k, delta)` / `lerp(1 - pow(k, delta))` (`player.gd:197,331`,
  `main.gd:3583,4211,5480`). Many amateur Godot projects get this wrong; this one
  never does.
- **Deliberate mobile performance engineering.** Quality tiers (`_apply_quality`,
  `main.gd:1385`) toggle shadows, bloom, every-other pearl light, plankton
  density and 3D resolution scale; animation culling pauses all creature
  `AnimationPlayer`s beyond 95/160 m on a 0.7 s cadence (`main.gd:6381`);
  flora gets `visibility_range` fades; the FFT ocean is gated off headless and
  behind a flag with a documented reason (`main.gd:668`).
- **Defensive resource handling.** `ResourceLoader.exists()` before every
  optional load, fallback primitive meshes when a GLB is missing
  (`main.gd:5386,5667`), a graceful pitched-voice fallback when a recorded line
  doesn't exist yet (`_say`, `main.gd:1335`), guarded JSON save parsing.
- **The hand-rolled soft-collision system fits the game.** Vertical cylinders +
  AABB walls with slide-along ejection (`player.gd:219–283`) is cheap, stable,
  tunable, and appropriate for a floaty swim game where real physics would fight
  the feel. The `solids` / `arena_solids` split (rebuilt per arena so colliders
  never leak) is clean and well-commented.
- **A real validation harness.** `probe.gd` (node census), `probe_games.gd`
  (drives all 5 minigames to a win through game state), `probe_load.gd`
  (save-restore assertion), `probe_l2.gd`, `probe_race.gd`, `probe_fetch.gd`.
  Headless-testable game logic is rare in hobby Godot projects and is the single
  best engineering decision here.
- **Comment quality.** Comments explain *intent and constraints* ("no collider on
  the wreck — it's an entrance, not a wall", `main.gd:6535`; the arena-ceiling
  rationale at `main.gd:2943`), not what the next line does.
- **Persistence is simple and robust**: one JSON dict, tolerant loading with
  defaults, written at every meaningful progress beat.

## 3. Confirmed bugs

Ordered by player impact. Line numbers are against the current `master`.

### B1 — The story intro never plays (dead feature)
`_build_intro()` (`main.gd:240`) builds the 4-panel Huluu/storm intro with
voice-over, and every probe defensively calls `_skip_intro()` — but **nothing
ever calls `_build_intro()`**. The game's narrative framing (who Huluu is, why
you collect pearls) is unreachable code. `first_session` (`main.gd:145`) is
computed in `_load_save()` and only used for hints.
**Fix:** at the end of `_ready()` (after `_load_save()`):
`if first_session: _build_intro()`. One line; all the machinery already works.

### B2 — Saved custom fish never respawn (load-order bug)
Crafting-studio fish are persisted (`custom_fish`, saved at `main.gd:1460`,
loaded at `main.gd:1435`) and spawned into the reef in
`_build_aquatic_creatures()` (`main.gd:1130`). But `_ready()` calls
`_build_aquatic_creatures()` *before* `_load_save()`, so on every relaunch the
spawn loop iterates an empty array. A child's crafted fish silently vanish
between sessions (crafted *friends* are fine — the lagoon is built on demand,
after loading).
**Fix:** move the `for cf in custom_fish` block into its own
`_spawn_custom_fish()` and call it from `_load_save()`, or load the save first
in `_ready()`.

### B3 — Chase camera stays zoomed-in after finishing the castle
`_enter_castle_interior()` tightens the camera (`cam_back = 6.5, cam_high = 4.2`,
`main.gd:2961`). `_exit_level2()` and `_return_to_courtyard()` restore
16.0/6.5 — but `_do_finish_level2()` (`main.gd:4032`), the path taken when the
child touches the Crown Star, does **not**. The player returns to the open ocean
with the indoor close-up camera for the rest of the session.
**Fix:** add `player.cam_back = 16.0; player.cam_high = 6.5` at the top of
`_do_finish_level2()`.

### B4 — Water-surface shader samples world UVs in view space
The overworld water fragment shader (`main.gd:654`) computes
`vec2 w = (MODEL_MATRIX * vec4(VERTEX,1.0)).xz`. In a Godot 4 spatial
**fragment** function, `VERTEX` is *view-space*, so this is not a world
position — the sparkle/caustic pattern is camera-dependent and slides across the
surface as the camera moves. The terrain caustics overlay does it correctly with
`INV_VIEW_MATRIX` (`main.gd:591`).
**Fix:** `vec3 wp = (INV_VIEW_MATRIX * vec4(VERTEX,1.0)).xyz; vec2 w = wp.xz;`

### B5 — The seek "color diamond" tap buttons never appear
`_melody_buttons_show()` is only ever called with `false` (`main.gd:5144`), so
the on-screen A/B/X/Y diamond promised by the design docs ("seek & melody = tap
the on-screen color diamond") is never shown. Touch players can still win seek by
swimming into the bush (`main.gd:6032`), so it's completable — but a documented
input path is dead code, and gamepad-only `_btn_pressed()` remains the only
button route.
**Fix:** call `_melody_buttons_show(true)` in the seek branch of `_start_game()`
(and hide on `_clear_game()`), or delete the diamond UI and update the docs.

### B6 — Wall-picture minigames retrigger on a timer, not on re-approach
`_tick_level2` (`main.gd:2467–2475`) opens a 2D minigame purely on proximity,
gated only by `mg_cool = 8.0` set on close. A child who stays near the picture
(likely — they just played there) gets the minigame force-reopened every 8
seconds. The secret chest solves this correctly with an arm/disarm pattern
(`secret_armed`, `main.gd:3640`), as does the portal (`portal_armed`).
**Fix:** copy the arm/disarm pattern: require leaving a >10 u radius before the
picture can fire again.

### B7 — Any mouse click throws the ball in fetch
`_tick_fetch` treats a global `Input.is_mouse_button_pressed(MOUSE_BUTTON_LEFT)`
as the throw input (`main.gd:6079`). Clicks consumed by UI still register —
tapping the pause gear (or anything else) mid-aim throws the ball, and with two
misses ends the game as a loss.
**Fix:** route the throw through `touch_ui.consume_action_just()` (already
implemented and currently unused, `touch_ui.gd:124`) plus `_unhandled_input`,
instead of raw polling.

### B8 — `probe_trial.gd` has rotted against the current minigames
The "child-paced playtest" bot still drives the **old** melody (Simon-says
`g["seq"]` / `g["input_i"]` / `stage == "input"`) and the old race
(`g["rings"]`) — none of those keys exist in the current orb-catch melody or
checkpoint play-place. Those branches silently no-op, so the pacing log the
audits rely on ("emulated 1-hour playthrough") wins those games only by
accident of the magnet assists.
**Fix:** update the melody branch to steer at un-caught orbs and the race branch
to steer at `g["checks"]`, or delete the probe if it's no longer trusted.

### B9 — Minor / hygiene
- **Unbounded arrays:** `pearl_lights` grows with an entry per respawned pearl
  forever (`main.gd:1184`); freed lights stay as invalid refs (guarded, but the
  array only grows). Same pattern for `anim_cull` (cleaned) vs `pulse_lights`
  (never cleaned).
- **Duplicated world constants:** `WATER_TOP`/`WORLD_R` defined in both
  `main.gd:5–6` and `player.gd:4–5`. They will drift someday.
- **Fixed-pixel HUD:** `hud_msg` sits at `y = 630` (`main.gd:1266`) and the
  intro/melody UI at hard-coded 1280×720 coordinates without anchors. With
  `expand` stretch on a 4:3 tablet the message floats mid-screen. The 2D
  minigame stage (`_mg2d_open`) does this correctly (scale + centre); the HUD
  should use the same approach or anchors. Also the `_mg2d_open` scale is
  computed once — a mid-game window resize/rotation breaks layout.
- **Melody is a near-auto-win:** the catch box is ±14×±7×±14 units
  (`main.gd:6262`) while orbs live in a ±16×±12 region — drifting a couple of
  units from spawn catches several orbs at once. If intentional for the
  audience, fine, but it makes the 30 s fail timer effectively unreachable and
  the game ~10 seconds long.

## 4. Structural weaknesses (not bugs, but the expensive kind of debt)

1. **`main.gd` is a 7,010-line god object.** World generation, 9 minigames, the
   Sky Lagoon, the castle interior, three overlay editors (craft/wardrobe/
   dress-up), save, audio, HUD, pause, cutscenes and the finale all share one
   class and two untyped scratch dictionaries (`g`, `mg`). Every minigame can
   read and corrupt every other's state; B8 happened precisely because nothing
   pins the shape of `g`. The clean seams already exist (`_start_game` /
   `_tick_*` / `_end_game`): each minigame could be a `RefCounted` subclass with
   `build/tick/teardown` in its own file with ~zero behaviour change. That single
   refactor would cut most future regression risk.
2. **Stringly-typed state machines.** `game`, `mg_kind`, `g["phase"]` are
   free-form strings compared inline. A typo (`"boss_leves"`) fails silently at
   runtime. Enums or at least class constants would make these checkable.
3. **Input polling is copy-pasted five times.** The same
   keyboard+joystick+touch-stick read block appears in `player.gd:159`,
   `_tick_slide` (`main.gd:5442`), `_tick_fairyshoot` (`main.gd:5728`),
   `_tick_dolls` (`main.gd:6185`) and fetch. None of it uses Godot's `InputMap`,
   so nothing is rebindable and every new scheme means editing five call sites.
   One `func read_move_input() -> Vector2` (or proper actions) fixes this.
4. **Dead code is accumulating.** Never called: `_build_fish` (`main.gd:6501`),
   `_build_megafauna` (`main.gd:6743`) — which also leaves `_tick_movers` and
   the `fish_schools` loop in `_tick_life` spinning over empty arrays every
   frame — `_flash_speaker_icon`, the `guide_fish` var, tiara/tail branches in
   the shop (`main.gd:4702–4747`, `_tick_shop`) that can never trigger because
   `SHOP_ITEMS` only contains beans, a one-entry `SKINS` wardrobe, and the
   entire 2.2 MB oceanfft addon behind a permanently-false flag. Each is small;
   together they make the file materially harder to audit than it needs to be.
5. **Per-instance material churn.** `_dress_nature` (`main.gd:1853`) allocates a
   fresh `StandardMaterial3D` per surface per nature instance (hundreds in the
   lagoon). `_aq_mat` shows the right pattern (cache by key); nature dressing
   should use the same cache keyed on `(greenish, col)`.
6. **Save format has no version field.** `_write_save` (`main.gd:1454`) writes a
   flat dict; any future rename silently loses fields. One `"v": 1` key plus a
   migration branch is cheap insurance. Also `_write_save` runs synchronously on
   every pearl pickup — harmless today, but batching to a 1 s debounce is one
   `Timer`.
7. **Asset weight:** `assets/terrain` alone is 61 MB of JPG/PNG source. Mobile
   export re-compresses to ETC2/ASTC, but the APK/AAB will carry large textures;
   several appear to be 2K where the triplanar tiling scale means 1K would be
   indistinguishable on the target device.

## 5. Recommended fix order

| Priority | Item | Effort |
|---|---|---|
| P0 | B1 intro never plays (1 line) | trivial |
| P0 | B2 custom-fish load order | trivial |
| P0 | B3 camera not restored after castle | trivial |
| P0 | B4 water shader view-space UV | trivial |
| P1 | B7 fetch fires on any click → use consume_action_just | small |
| P1 | B6 wall-picture arm/disarm | small |
| P1 | B5 seek diamond buttons (show or delete + doc fix) | small |
| P1 | B8 refresh probe_trial to current game shapes | small |
| P2 | Dead-code sweep (§4.4) — do before any refactor | small |
| P2 | Extract minigames from main.gd behind the existing start/tick/end seams | medium |
| P2 | Single input-read helper (or InputMap actions) | small-medium |
| P2 | Anchor-based HUD; share the _mg2d_open scaling for overlays | small |
| P3 | Material cache in _dress_nature; save versioning; texture size pass | small |

The four P0 items are one-or-two-line fixes that restore intended, already-built
behaviour — the intro, crafted-fish persistence, the camera, and the water look.
They should go in before anything else.
